# 有序事件多线程竞态修复设计

## 问题描述

### 背景

`CommonEventControlManager` 中的有序事件处理涉及多个线程并发访问共享数据：

| 线程 | 执行的函数 |
|------|-----------|
| IPC/Service 线程 | `EnqueueOrderedRecord`, `FinishReceiverAction`, `GetMatchingOrderedReceiver` |
| ffrt `orderedQueue_` 串行线程 | `ProcessNextOrderedEvent`, `CurrentOrderedEventTimeout(true)` |
| Dump 线程 | `GetOrderedEventRecords` |

### 发现的竞态条件

#### 竞态 #1 (Critical): `CurrentOrderedEventTimeout` 无锁访问 `orderedEventQueue_`

**位置**: `common_event_control_manager.cpp:670,676`

```cpp
// 整个函数未持有 orderedMutex_
if (orderedEventQueue_.empty()) {        // 无锁读
    return;
}
sp = orderedEventQueue_.front();         // 无锁读
```

**后果**: `std::vector::emplace_back` 可能触发 reallocation，此时另一个线程读取 `empty()` 或 `front()` 是未定义行为，可直接导致 crash。

#### 竞态 #2 (Critical): `CurrentOrderedEventTimeout` 和 `FinishReceiverAction` 同时写 `OrderedEventRecord`

**位置**: `common_event_control_manager.cpp:687-704` vs `:731-740`

当超时发生且慢速 subscriber 刚好同时响应时，两个线程并发修改同一对象的相同字段：
- `sp->state`
- `sp->curReceiver`
- `sp->commonEventData` code/data
- `sp->resultAbort`

#### 竞态 #3 (Critical): `ProcessNextOrderedEvent` 释放锁后继续写 `sp`

**位置**: `common_event_control_manager.cpp:602-608`

```cpp
}  // orderedMutex_ 在此释放
sp->nextReceiver++;                     // 无锁写
SetTime(recIdx, sp, ...);               // 写 sp->receiverTime
NotifyOrderedEvent(sp, recIdx);         // IPC阻塞调用 + 写 sp
```

`NotifyOrderedEvent` 内部的 `receiver->NotifyEvent()` 是同步 IPC 调用，会阻塞。在此期间 subscriber 的 `finishReceiver` 回调到达，`FinishReceiverAction` 会写相同的 `sp` 字段。

## 解决方案设计

### 核心思路

1. **队列访问保护**: `orderedEventQueue_` 的所有访问（读/写）必须在 `orderedMutex_` 保护下
2. **记录字段保护**: `OrderedEventRecord` 的字段修改需要独立的锁保护
3. **原子状态**: `state` 字段使用 `std::atomic` 简化并发读写

### 设计变更

#### 1. `OrderedEventRecord` 结构体变更

```cpp
struct OrderedEventRecord : public CommonEventRecord {
    // 原有字段...
    
    std::atomic<int8_t> state;           // 改为原子变量
    ffrt::mutex recordMutex_;            // 新增：per-record 锁
};
```

**设计理由**:
- `state` 字段被多个线程频繁读写，使用原子操作避免锁开销
- 其他字段（`curReceiver`, `resultAbort`, `commonEventData` 等）通过 `recordMutex_` 保护

#### 2. 锁策略

| 数据 | 保护方式 |
|------|----------|
| `orderedEventQueue_` (容器) | `orderedMutex_` |
| `OrderedEventRecord` 字段 | `recordMutex_` (per-record) |
| `state` 字段 | `std::atomic<int8_t>` |

#### 3. IPC 调用锁释放策略

`NotifyOrderedEvent` 中的 IPC 调用 (`NotifyEvent`) **不持有锁**，避免：
- Binder 线程池耗尽时可能死锁
- 长时间阻塞导致队列无法处理

## 函数重构

### 重构原则

- 行宽不超过 120 字符
- 函数不超过 50 行
- 拆分为职责单一的帮助函数

### 重构结果

| 原函数 | 原行数 | 新行数 | 新增帮助函数 |
|--------|--------|--------|--------------|
| `CurrentOrderedEventTimeout` | 57 | 23 | `GetFrontOrderedRecord`, `CheckAndRescheduleTimeout`, `HandleTimeoutReceiver` |
| `ProcessNextOrderedEvent` | 85 | 30 | `ProcessOrderedEventQueueLocked`, `CheckTimeoutForceReceive`, `HandleFinalSubscriber`, `PrepareNextReceiverRecord` |
| `NotifyOrderedEvent` | 66 | 34 | `NotifyFrozenSubscriber`, `PrepareOrderedNotify`, `HandleOrderedNotifyResult`, `LogOrderedNotifySuccess` |

## 风险评估

### 低风险

- `std::atomic` 使用正确，无 ABA 问题
- `ffrt::mutex` 与 `std::lock_guard` 配合使用，无死锁风险
- IPC 调用前后释放锁，避免 Binder 死锁

### 注意事项

- `recordMutex_` 是 per-record 锁，不同记录可并行处理
- `orderedMutex_` 保护队列容器，不保护记录字段
- `state` 原子操作与 `recordMutex_` 可并行，但 `state.load()` 后的判断仍需业务逻辑保证正确性
